# AgentHub AI Collaboration Record

本文档用于答辩和验收，说明 AgentHub 在开发过程中如何使用 AI 协作能力完成需求理解、任务拆解、代码生成、UI 调整、验证和交付。课题评分中 AI 协作能力占比最高，因此这里重点记录“如何协作”，而不是只描述最终界面。

## 1. 背景与目标

课题要求建设一个简化版多 Agent 协作平台，核心体验接近 IM 聊天：用户可以创建单聊和群聊，会话中能出现部署状态卡片、网页、文档、PPT、代码 Diff 等产物，并由一个主 Agent 负责理解需求、拆解任务和调度多个子 Agent。

AgentHub 的实现目标是：

- 用 PMO 主 Agent 呈现“需求理解 -> 任务拆解 -> 子 Agent 并行执行 -> 失败降级 -> 产物回收”的完整链路。
- 接入 Codex、Claude Code 等主流 Agent 平台入口，并允许用户创建自己的 Agent。
- 将 Agent 产物放入聊天上下文和右侧工作台中，支持预览、编辑、Diff、版本历史和部署。
- 让答辩时可以通过一个 3 分钟演示清楚证明功能覆盖，而不是只展示静态页面。

## 2. 协作规则

项目采用“AI 参与但人类可控”的协作方式：

- 先写 Spec 和验收口径，再进入 UI 或代码实现，避免只做视觉堆叠。
- 每次改动都保持小步提交，并推送到 Git 仓库，方便回溯。
- `.env`、`.env.local`、真实 API Key、Endpoint 等敏感信息只放本地环境变量，不进入 Git。
- 生成代码后必须经过静态检查、类型检查或场景冒烟验证，至少保证本轮改动不会破坏核心路径。
- UI 参考素材只提炼风格原则，不直接复制视觉资产；AgentHub 保留自己的机器人品牌和蓝紫主色。
- 对答辩要求按“功能覆盖证据”组织，不只按组件文件组织。

## 3. Agent 分工

| 角色 | 定位 | 在项目中的体现 |
| --- | --- | --- |
| PMO / Main Agent | 类似 PM 或 PMO，负责理解用户目标、拆解任务、派发子任务、汇总风险 | 群聊会话、任务流、右侧任务面板、协作记录 |
| Codex | 代码实现、重构、Diff、测试修复 | 代码编辑、Diff 卡片、版本历史、仓库提交 |
| Claude Code | 架构分析、冲突复核、降级接管 | 冲突处理、代码审查、架构说明 |
| UX Reviewer | 用户自建 Agent 示例，负责 UI 反馈 | 自建 Agent、聊天成员、体验优化建议 |

## 5. 验证方式

开发中使用以下方式验证质量：

- `npm run lint`：检查前端和共享包的基础代码规范。
- `npm run typecheck`：检查 TypeScript 类型。
- `npm run test -- --runInBand`：运行 Jest 测试。
- `npm run smoke:acceptance`：验证核心页面、后端健康检查和关键入口可用。
- 浏览器手动检查：登录页、聊天页、会话列表、右侧产物面板、移动端宽度。

文档类改动不改变运行逻辑，主要通过 `git diff --check` 和人工阅读验证格式与内容。

## 6. 可向评审说明的亮点

- AgentHub 不是简单聊天 UI，而是把聊天、任务调度、产物和部署放在同一个工作台中。
- PMO 主 Agent 的调度过程可视化，能说明复杂任务如何被拆解和派发。
- 群聊不是普通多人聊天，而是多 Agent 协作空间，成员有能力标签和工作状态。
- 产物不只作为附件出现，而是可以在右侧工作台继续预览、编辑、对比和部署。
- 项目记录了 AI 如何参与需求分析、UI 迭代和代码交付，能对应课题“AI 协作能力”的评分项。

## 7. 当前限制与后续计划

- Codex / Claude Code 的真实可用性依赖服务器上的 CLI 与密钥配置，界面会明确展示“可用”或“待配置”状态。
- 消息操作已经具备上下文管理基础，后续可继续增强回复、引用、重新生成、锁定上下文等更完整的 IM 操作。
- 代码冲突处理目前重点展示流程和 Diff，后续可接入更真实的 Git 冲突检测与自动合并策略。
- 文档段落引用到 Agent 的链路已经有产品形态，后续可强化选择文本后的快捷派发体验。
