# AgentHub AI Collaboration Record

本文档用于记录 AgentHub 开发过程中的 AI 协作方式，说明项目如何通过人机协作完成需求理解、任务拆解、代码生成、UI 调整、验证和交付。文档重点记录“如何协作”，而不是只描述最终界面。

## 1. 背景与目标

课题要求建设一个简化版多 Agent 协作平台，核心体验接近 IM 聊天：用户可以创建单聊和群聊，会话中能出现部署状态卡片、网页、文档、PPT、代码 Diff 等产物，并由一个主 Agent 负责理解需求、拆解任务和调度多个子 Agent。

AgentHub 的实现目标是：

- 用 PMO 主 Agent 呈现“需求理解 -> 任务拆解 -> 子 Agent 并行执行 -> 失败降级 -> 产物回收”的完整链路。
- 接入 Codex、Claude Code 等主流 Agent 平台入口，并允许用户创建自己的 Agent。
- 将 Agent 产物放入聊天上下文和右侧工作台中，支持预览、编辑、Diff、版本历史和部署。
- 通过真实功能链路展示多 Agent 协作从需求到产物交付的完整过程。

## 2. 协作规则

项目采用“AI 参与但人类可控”的协作方式：

- 先写 Spec 和功能边界，再进入 UI 或代码实现，避免只做视觉堆叠。
- 每次改动都保持小步提交，并推送到 Git 仓库，方便回溯。
- `.env`、`.env.local`、真实 API Key、Endpoint 等敏感信息只放本地环境变量，不进入 Git。
- 生成代码后必须经过静态检查、类型检查或场景冒烟验证，至少保证本轮改动不会破坏核心路径。
- UI 参考素材只提炼风格原则，不直接复制视觉资产；AgentHub 保留自己的机器人品牌和蓝紫主色。
- 对项目要求按“功能覆盖证据”组织，不只按组件文件组织。

## 3. Agent 分工

| 角色 | 定位 | 在项目中的体现 |
| --- | --- | --- |
| PMO / Main Agent | 类似 PM 或 PMO，负责理解用户目标、拆解任务、派发子任务、汇总风险 | 群聊会话、任务流、右侧任务面板、协作记录 |
| Codex | 代码实现、重构、Diff、测试修复 | 代码编辑、Diff 卡片、版本历史、仓库提交 |
| Claude Code | 架构分析、冲突复核、降级接管 | 冲突处理、代码审查、架构说明 |
| UX Reviewer | 用户自建 Agent 示例，负责 UI 反馈 | 自建 Agent、聊天成员、体验优化建议 |

### 4.1 人机协作时间线

这一部分记录的不是最终功能清单，而是开发者与 AI 在项目推进中的真实协作方式：用户持续提出目标、体验反馈和现场问题，AI 负责拆解、实现、验证、提交，并在每轮结果后根据反馈继续调整。

| 协作轮次 | 用户输入 / 反馈 | AI 处理方式 | 形成的结果 |
| --- | --- | --- | --- |
| 课题理解 | 用户提供课题要求，说明需要对话列表、单聊、群聊、部署卡片、主 Agent、多 Agent、产物预览编辑等能力 | AI 将题面拆成产品模块、需求项和开发优先级 | 明确 AgentHub 要做成 IM 式多 Agent 协作工作台 |
| UI 风格收敛 | 用户指出“改完的 UI 有很大一股 AI 味”，并提供机器人品牌图和参考视频 | AI 提炼成熟 SaaS 工作台风格，减少大渐变、大卡片和宣传页式布局 | 登录页、会话列表、右侧工作台逐步收敛为简洁、轻量、专业的界面 |
| 会话体验修复 | 用户反馈消息工具栏 hover 突兀、代码卡片过小、生成内容突然全部出现、切换群聊后消息串台 | AI 检查消息渲染、流式输出、代码卡片、任务状态、会话同步和工作台同步逻辑 | 优化消息操作、代码卡片展示、任务状态条、产物工作台与当前会话的联动 |
| 产物与部署链路 | 用户希望小型网站能预览、编辑、部署，并接入自己的阿里云服务器 | AI 设计自托管部署配置、SSH 密钥写入、默认服务器、部署状态卡片和部署面板 | 支持默认服务器静态部署、部署进度、日志、访问链接和失败提示 |
| 知识库与文件 | 用户询问知识库和文件栏用途，并指出知识库乱码、缺少内容 | AI 明确二者边界：文件沉淀上传与产物，知识库沉淀可检索片段 | 完善知识库、文件中心和 Agent 引用资料的产品逻辑 |
| 工作流联动 | 用户认为独立工作流页面输入输出不直观，希望在会话输入框引用工作流 | AI 调整工作流引用思路，让工作流作为会话上下文能力，而不是孤立页面 | 会话输入框可引用已保存工作流，并带入当前会话任务 |
| 多用户群聊 | 用户提出真实用户群聊中 Agent 应该可启用/静音，并只读取启用区间 | AI 将群聊协作改为“真人自由讨论 + 群主启用 Agent + 上下文区间过滤”的方案 | 群聊在 2 个及以上真实用户时出现智能体启用/静音和上下文过滤能力 |
| 自建智能体 | 用户希望自建智能体能接入自己的 LLM API，并能加入群聊执行任务 | AI 完善自建 Agent 配置项，包括模型、Base URL、API Key、提示词、能力标签和群聊成员联动 | 自建智能体不再只是联系人展示，而是具备可配置执行边界 |
| 移动端定位 | 用户指出手机端不需要完整工作台，应像遥控器，电脑端才是主执行终端 | AI 调整移动端定位，只保留 AI 助手、消息发送、会话查看和任务确认 | 桌面端负责编辑、预览、部署，手机端负责轻量确认和沟通 |
| 真实能力清理 | 用户要求确保项目能力真实存在，不展示虚构能力 | AI 扫描不存在的平台展示、假部署 URL、无密钥成功兜底等风险点 | 保留 Codex / Claude Code / 自建 Agent 真实入口；未配置工具直接返回明确失败 |
| 提交与隐私 | 用户要求每次改动推送，并确认 `.env` 不会提交 | AI 每轮提交前检查暂存区、隐私关键词、`.gitignore` 和远端状态 | 代码推送到 Gitee / GitHub，真实密钥、SSH 私钥、部署输出和日志不进入仓库 |
| 文档归档 | 用户要求产品文档、技术文档和 AI 协作记录都能直接提交 | AI 按当前仓库功能重新同步文档，去掉旧称呼和过时功能描述 | `docs/SPEC.md`、`docs/ARCHITECTURE.md`、`docs/AI_COLLABORATION.md`、`docs/FINAL_DELIVERY.md` 与当前实现一致 |

### 4.2 协作证据说明

- 对话记录中保留了用户的连续反馈，例如“UI 有 AI 味”“代码窗口太小”“部署卡片应在同一卡片更新”“工作台没有和会话同步”“不要展示虚构能力”等。
- 每个问题不是一次性生成，而是经过“用户发现问题 -> AI 制定修复计划 -> 修改代码 -> 本地验证 -> 用户继续反馈”的循环。
- 仓库提交记录可以对应关键节点，例如 UI 优化、部署链路、自建智能体、会话同步、移动端兼容和最终清理虚构能力。
- 验证过程包含 `lint`、`typecheck`、`test`、`build`、隐私扫描和 Git 远端推送检查。
- 对于未完成或依赖环境的能力，文档明确写出边界，例如 Codex / Claude Code 依赖服务器 CLI 与密钥配置，未配置时显示待配置或失败原因。

## 5. 验证方式

开发中使用以下方式验证质量：

- `npm run lint`：检查前端和共享包的基础代码规范。
- `npm run typecheck`：检查 TypeScript 类型。
- `npm run test -- --runInBand`：运行 Jest 测试。
- `npm run smoke:acceptance`：验证核心页面、后端健康检查和关键入口可用。
- 浏览器手动检查：登录页、聊天页、会话列表、右侧产物面板、移动端宽度。

文档类改动不改变运行逻辑，主要通过 `git diff --check` 和人工阅读验证格式与内容。

## 6. 项目亮点

- AgentHub 不是简单聊天 UI，而是把聊天、任务调度、产物和部署放在同一个工作台中。
- PMO 主 Agent 的调度过程可视化，能说明复杂任务如何被拆解和派发。
- 群聊不是普通多人聊天，而是多 Agent 协作空间，成员有能力标签和工作状态。
- 产物不只作为附件出现，而是可以在右侧工作台继续预览、编辑、对比和部署。
- 项目记录了 AI 如何参与需求分析、UI 迭代和代码交付，体现了人机协作在复杂软件项目中的实际作用。

## 7. 当前限制与后续计划

- Codex / Claude Code 的真实可用性依赖服务器上的 CLI 与密钥配置，界面会明确展示“可用”或“待配置”状态。
- 消息操作已经具备上下文管理基础，后续可继续增强回复、引用、重新生成、锁定上下文等更完整的 IM 操作。
- 代码冲突处理目前重点展示流程和 Diff，后续可接入更真实的 Git 冲突检测与自动合并策略。
- 文档段落引用到 Agent 的链路已经有产品形态，后续可强化选择文本后的快捷派发体验。
