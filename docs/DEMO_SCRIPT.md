# AgentHub 3-Minute Demo Script

本文档用于录制或现场演示 AgentHub。目标是在 3 分钟内覆盖课题要求：对话列表、单聊、群聊、PMO 主 Agent、多 Agent、产物预览编辑、Diff、版本历史、部署和上下文管理。

## 1. 演示前检查

```bash
npm run dev:all
```

打开本地地址后确认：

- 可以进入登录页和主工作台。
- 左侧导航和会话列表正常显示。
- 验收指南或演示入口可以注入 demo 数据。
- 右侧工作台可以切换 Preview、Code、Diff、Slides、History、Deploy、Context。
- 如果真实模型不可用，使用内置演示数据继续录制，不影响答辩主链路。

## 2. 时间轴

| 时间 | 画面 | 讲解 |
| --- | --- | --- |
| 0:00 - 0:20 | 登录页与品牌入口 | “这是 AgentHub，一个多 Agent 协作平台。它不是普通聊天机器人，而是把需求、Agent 调度、产物和部署放在同一个工作台里。” |
| 0:20 - 0:40 | 进入工作台，展示左侧导航和会话列表 | “左侧是 IM 式会话列表，支持单聊、群聊、置顶、归档、搜索和 Agent 能力标签。” |
| 0:40 - 1:05 | 切换 Codex 单聊和验收群聊 | “单聊适合让某个 Agent 专注执行，群聊适合复杂任务，由 PMO 主 Agent 统一协调。” |
| 1:05 - 1:30 | 展示 PMO 任务面板 | “PMO 会把用户需求拆成设计、代码、检查、部署等子任务，并派给 Codex、Claude Code、Open Code 或自建 Agent。” |
| 1:30 - 1:55 | 展示聊天中的产物卡片和右侧 Preview | “Agent 的回复不是静态文本，网页、文档、PPT、代码都会进入右侧工作台继续操作。” |
| 1:55 - 2:15 | 切到 Code、Diff、History | “代码可以编辑，Diff 可以对比和应用，版本历史可以回看 v1/v2，满足产物编辑和二次迭代要求。” |
| 2:15 - 2:35 | 切到 Deploy 面板 | “部署卡片展示构建、上传、发布、日志和访问链接，模拟部署到三方平台的完整状态。” |
| 2:35 - 2:50 | 展示 Context 和 My Agents | “用户可以把消息、文档段落或代码片段加入上下文，也可以创建自己的 Agent 参与协作。” |
| 2:50 - 3:00 | 回到验收指南或群聊总结 | “这条链路覆盖了课题中的交互体验、主 Agent、多 Agent 接入、产物预览编辑和部署能力。” |

## 3. 讲解重点

答辩时建议一直围绕三个关键词：

- 协作：PMO 负责拆解和协调，子 Agent 负责执行。
- 产物：网页、文档、PPT、代码、Diff、版本和部署都能在工作台内继续操作。
- 可验收：每个课题要求都有明确入口和可展示证据。

## 4. 备用讲法

如果现场模型接口失败：

> “这里使用内置验收演示数据继续展示。真实模型接入通过统一 adapter 层完成，接口不可用不会影响产品主流程，因为 PMO 调度、产物工作台、部署状态和上下文管理都可以独立验证。”

如果被问到 Open Code 是否真实执行：

> “当前版本中 Codex 和 Claude Code 已进入适配器设计，Open Code 在联系人和协作流程中展示为主流 Agent 平台，后续只需要按同一 adapter interface 补齐执行器，不需要改聊天和工作台产品逻辑。”

如果被问到 API Key：

> “密钥只通过本地 `.env` 或部署平台密钥配置，仓库提交中不会包含真实 Key。代码和文档只保留环境变量名。”

## 5. 录屏建议

- 浏览器宽度建议保持 1440px 以上，三栏结构更清楚。
- 鼠标动作放慢，不要频繁来回切换。
- 每个面板只停留 3 到 5 秒，重点讲“它证明了哪条课题要求”。
- 录制前关闭无关浏览器标签和通知。
- 如果使用真实接口，先准备一条短需求，避免等待时间过长。

## 6. 结尾总结

推荐最后一句：

> “AgentHub 的核心不是把多个 Agent 名字放在一起，而是让它们在一个 IM 式工作台里完成从需求、调度、产物、编辑到部署的闭环。”
