## 2. 飞书主文档建议结构

飞书文档建议只放评委最需要的信息，不要堆太多开发细节。仓库里的 Markdown 可以作为补充链接。

### 2.1 项目简介

AgentHub 是一个面向多 Agent 协作的软件生成工作台。项目以 IM 会话为入口，支持单聊、群聊、主 Agent 调度、多 Agent 协作、产物预览编辑、代码 Diff、版本历史、上下文引用和部署状态展示。

一句话总结：

> AgentHub 让用户像在群聊里协作一样，把需求交给 PMO 主 Agent 拆解，再调度 Codex、Claude Code 和自建 Agent 完成从需求到产物、预览、编辑和部署的闭环。

### 2.2 核心使用流程

1. 用户登录 AgentHub，进入工作台。
2. 用户创建单聊或群聊，并邀请真实用户或智能体成员。
3. 群聊中默认允许真人自由讨论，智能体处于静音状态。
4. 群主确认需求后启用智能体，PMO 主 Agent 读取启用区间内的上下文。
5. PMO 拆解任务，并把子任务派给代码、设计、测试或部署 Agent。
6. Agent 在消息流中输出说明文字，代码进入代码卡片，产物进入右侧工作台。
7. 用户在产物工作台查看预览、代码、Diff、PPT、历史、部署和上下文。
8. 用户可引用消息、文档段落或代码片段，继续交给指定智能体处理。
9. 用户确认产物后触发部署，系统展示部署状态和访问链接。

### 2.3 课题要求对应关系

| 课题要求 | AgentHub 对应实现 |
| --- | --- |
| 对话列表 | 左侧会话列表支持搜索、置顶、归档、单聊和群聊标签 |
| 单聊模式 | 支持 AI 智能助手、Codex、Claude Code、自建 Agent 单聊 |
| 群聊模式 | 支持真实用户 + PMO 主 Agent + 多个执行 Agent 协作 |
| 部署状态卡片 | 消息流和产物工作台展示部署中、成功、失败和访问链接 |
| 主 Agent 概念 | PMO 主 Agent 负责理解需求、拆解任务、调度和汇总 |
| 多 Agent 接入 | 展示 Codex、Claude Code 和用户自建 Agent |
| 自建 Agent | 用户可创建自定义名称、能力标签、提示词和 LLM API 配置 |
| 产物预览与编辑 | 支持网页预览、代码编辑、Diff、版本历史、PPT/文档展示 |
| 二次交互 | 支持消息引用、上下文添加、交给群内指定智能体继续处理 |
| 多用户协作 | 群聊中 2 个及以上真实用户时提供智能体启用/静音和上下文过滤 |

### 2.4 技术架构简介

| 层级 | 说明 |
| --- | --- |
| 前端 | Next.js 16、React 19、TypeScript、Tailwind CSS、Zustand |
| 会话通信 | WebSocket 负责实时消息、Agent 状态和任务状态同步 |
| Agent 适配 | 通过统一适配层连接 OpenAI 兼容模型、Claude Code、Codex CLI 或自建 LLM API |
| 产物工作台 | 将 Agent 回复中的代码、文档、PPT、预览 URL、部署状态转为可操作产物 |
| 部署链路 | 支持默认服务器静态部署，并保留 Vercel / 自托管扩展路径 |
| 数据与状态 | 前后端共享类型放在 `packages/shared`，核心业务状态由 store 和服务端接口协同维护 |

更详细的技术说明见 `docs/ARCHITECTURE.md`。

### 2.5 AI 协作开发流程

结题文档中建议重点说明“怎么和 AI 协作”，而不是只说“用了 AI 写代码”。

可以按下面这条线讲：

1. 先分析课题背景，整理评分点和核心验收路径。
2. 将需求拆成产品模块：会话、主 Agent、多 Agent、产物工作台、部署、上下文管理。
3. 使用 AI 辅助生成方案和代码，但每轮改动都经过本地运行、类型检查、构建和人工验收。
4. UI 设计经过多轮调整，从“AI 炫技感”收敛为“成熟 SaaS 工作台”。
5. 对会话串台、消息不回复、产物不同步、部署状态分裂、移动端兼容等问题逐步排查修复。
6. 将关键协作记录沉淀到 `docs/AI_COLLABORATION.md`，用于说明 AI 参与了需求分析、架构设计、代码实现、测试验证和交付整理。

### 2.6 Demo 录屏建议

Demo 不需要把所有功能点都点一遍，重点是展示完整闭环。

| 时间 | 展示内容 | 讲解重点 |
| --- | --- | --- |

## 3. 本地运行方式

```bash
docker compose up -d postgres redis
npm run dev:all
```

默认访问地址：

- 前端工作台：`http://localhost:3000`
- 后端健康检查：`http://localhost:3002/api/health`

常用验证命令：

```bash
npm run lint
npm run typecheck
npm run build
npm run smoke:acceptance
```

## 4. 隐私与提交说明

提交仓库时必须确认：

- 不提交 `.env.local`、`.env`、真实 API Key、SSH 私钥。
- 不提交部署输出、数据库文件、日志、临时截图。
- 仓库只保留 `.env.example` 中的环境变量名称和示例说明。
- 真实密钥放在本地环境变量或部署平台密钥配置中。

## 5. 答辩备用说明

如果模型接口临时不可用：

> AgentHub 的核心是多 Agent 协作产品链路，模型通过统一适配层接入。若某个模型接口临时不可用，系统会展示待配置或失败原因；已有会话、产物工作台、上下文管理和部署状态仍能说明完整闭环。

如果被问到 GitHub 地址但当前使用 Gitee：

> 当前代码托管在 Gitee 仓库。如果提交平台强制要求 GitHub，可以将同一仓库镜像到 GitHub；代码内容、提交历史和隐私规则保持一致。

如果被问到真实部署能力：

> 当前自托管服务器用于轻量静态产物发布。正式生产环境可以把前端、后端、数据库、对象存储和部署目标拆分，并通过平台密钥管理环境变量。
